# 統合基盤への移行プラン

## 現状 (2026-07-10時点)

- Instagram自動投稿を `ig_akiya`, `ig_post`, `ig_insight` などリポジトリ単位で運用中(現在3アカウント分)。
- 各リポジトリは `scripts/post.py` / `scripts/insight.py` をコピーして持っており、既に実装が分岐している。
  - `ig_akiya`: `graph.facebook.com/v22.0`
  - `ig_post`: `graph.instagram.com/v25.0`
  - 同じバグ(コンテナステータス取得時に存在しない `error_message` フィールドを要求し400になる)が `ig_akiya` では修正済みだが `ig_post` には残っている。
- トークンはリポジトリごとにGitHub Secretsで個別管理(`INSTAGRAM_ACCESS_TOKEN`など)。60日ごとに手動で長期トークンへ変換・再設定が必要。

## 移行を検討するトリガー

- 4アカウント目を追加するタイミング、または
- 「同じ修正を複数リポジトリに横展開する」作業が発生した時点(既に発生している)

どちらか早い方で着手する。

## 統合基盤の方向性

- アカウント情報(トークン、IG Account ID、投稿スケジュール、コンテンツ格納場所)をDBのレコードとして管理し、1つのアプリ/ワーカーが全アカウント分をループ処理する構成にする。
- API呼び出しロジック(コンテナ作成・ステータス確認・publish・トークン更新)は1箇所に集約し、アカウント差分は設定値のみで吸収する。
- トークンの長期化・失効管理を自動化するバッチを組み込み、手動での `exchange_token.py` 実行をなくす。

## 未確定事項 (要検討)

- コンテンツ(動画ファイル)の管理方法: リポジトリ内配置を続けるか、外部ストレージ(S3/R2等)に移すか。
- 投稿スケジュールの実行環境: GitHub Actions cronを使い続けるか、常駐ワーカー/サーバーレスに移すか。
- マルチテナントのDB設計(1レコード1アカウント、または顧客単位でのグルーピングが必要か)。
